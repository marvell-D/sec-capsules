# v0.1 初始纵切规范（已完成）

本文件保留 v0.1 的原始目标，用来解释项目为什么先选择 `httpx`、`katana`、`nuclei`，以及它刻意不做什么。它是历史设计边界；真实执行相关的能力由 v0.1.1 补齐，当前完整状态以[中文开发者手册](zh-CN/V0.1.1_开发者手册.md)为准。

## 1. v0.1 要证明的命题

当上层 Agent 已经决定调用某个安全工具时，是否可以把这次调用标准化成下列可组合的生命周期：

```text
工具卡发现 -> argv 计划 -> Scope 判定 -> 原始证据 -> 专属 parser
    -> 统一对象 -> token 受限 ObservationPacket
```

重点不是“让 Agent 能扫描更多目标”，而是“让既有工具的调用知识、结果与模型上下文有清晰边界”。

## 2. v0.1 交付能力

| 能力 | v0.1 的实现位置 | 目的 |
|---|---|---|
| 加载内置 Capsule | `core/registry.py` | 由文件系统发现工具卡，避免中心注册表。 |
| 生成安全命令计划 | `core/planner.py` | 从 profile 模板生成 argv，而非 shell 字符串。 |
| 执行前 Scope 判定 | `core/scope.py` | 将目标、活动类型、速率与审批规则集中处理。 |
| 原始输出存证 | `core/artifacts.py` | 保存 artifact 与 SHA-256，保留可追溯证据。 |
| 结构化解析 | `capsules/*/parser.py` | 将不同工具 JSONL 映射到 service/endpoint/finding/evidence。 |
| 低 token 观察包 | `core/observation.py` | 让 Agent 默认看到摘要，不被原始输出淹没。 |
| CLI 与 MCP | `interfaces/` | 用两个薄接口复用同一套 Core 语义。 |

v0.1 使用 `httpx` 证明 HTTP 服务发现、`katana` 证明 URL/端点收集、`nuclei` 证明 finding 输出；三个工具共同覆盖 WebSec 流程里最基础的三种输出形态。

## 3. v0.1 明确不做的事情

- 不根据扫描结果自主选择下一个工具。
- 不进行漏洞利用、凭据攻击或攻击链规划。
- 不绕过 Scope、审批或目标授权。
- 不把 Nuclei、Katana 等工具重新实现为 Python 扫描器。
- 不承诺所有外部工具都能在不受信任环境安全运行。

这些限制不是功能缺失，而是项目的产品边界。上层 Agent 可以有自己的规划能力，但它应经由 Capsule Runtime 的明确调用边界使用工具。

## 4. v0.1.1 如何完成这条纵切

v0.1 的最初设计更多覆盖 fixture、计划与数据流。v0.1.1 增加了 binary preflight、真实子进程控制、DNS-aware Scope、审批审计、静态 Recipe DAG、MCP execution gate 与本机 Juice Shop E2E，使这条纵切可以对自有靶场的真实调用进行验证。详细保证与限制见[V0.1.1 执行门槛](V0.1.1_EXECUTION_GATE.md)。
