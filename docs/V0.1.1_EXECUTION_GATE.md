# v0.1.1 Execution Gate

v0.1.1 completes the original vertical slice before the project broadens to `nmap` and `ffuf`.

## Runtime Guarantees

- Runtime commands are argv lists and never pass through a shell.
- Live execution preflights the declared executable and records its resolved path and version output.
- Child processes receive no stdin, run in an isolated process group, and are terminated as a group on timeout.
- Stdout and stderr are independently capped. The run manifest records truncation rather than silently discarding it.
- A missing binary, non-zero exit, or timeout produces a terminal run status and preserves available evidence.

## Scope And Approval

- The runtime normalizes target URLs and enforces include/exclude rules before execution.
- A live run resolves the target hostname and applies the private/link-local/metadata policy to every returned address.
- A capsule profile cannot request a higher rate limit than the scope permits.
- `require_approval_for` and profile `requires_approval` require an approval record with an id, operator identity, allowed actions, and target bindings.

Approval records are local audit records. They make a human acknowledgement explicit in the execution trace; they do not prove target ownership or replace an organization's authorization process.

## Context Boundary

- Raw output is persisted as an artifact and remains outside normal model observations.
- Artifact reads require `artifact://` references rooted inside a known run, with line and character caps.
- Recipes validate a static dependency graph. Recipe observations contain step summaries and references, not every structured result or raw artifact.

## E2E Boundary

`scripts/e2e-local.sh` runs only against the repository's local Juice Shop compose environment. The GitHub Actions E2E workflow is manual-only and runs on the dedicated self-hosted runner.
