# v0.1.1 执行门槛（已完成）

v0.1.1 的目标是在扩展到 `nmap`、`ffuf` 前，先让 v0.1 的 WebSec 垂直切片具备明确、可测试的真实执行边界。本文件只描述已实现的保证和限制；完整逐文件解释见[中文开发者手册](zh-CN/V0.1.1_开发者手册.md)。

## 1. 外部命令 Runtime 保证

实现位于 `src/sec_capsules/core/executor.py` 与 `core/runner.py`。

- CommandPlan 是 argv token 列表，从不经 shell 解释。
- 真实执行前 `inspect_tool()` 定位 binary、执行 version command，并把路径和版本写入结果。
- 子进程 stdin 指向 `DEVNULL`，且只继承受控的环境变量 allowlist。
- `start_new_session=True` 创建进程组；超时后先发送 `SIGTERM`，宽限后 `SIGKILL` 整个组。
- stdout 与 stderr 分别由读取线程持续排空并各自限制保留字节数，避免 pipe 写满死锁或无限占用内存。
- 缺 binary、非零退出或超时都会成为终态，并尽力保留已有 stdout/stderr artifact，而不是直接丢失上下文。

`RunResult.status` 的主要取值为：

| 状态 | 含义 |
|---|---|
| `dry_run` | 没有 `--execute`，也没有 fixture；只生成计划与记录。 |
| `replayed` | 使用 fixture 解析，不启动外部二进制。 |
| `preflight_failed` | binary 或 version command 不能正常使用。 |
| `succeeded` | 工具退出码为 0，且未超时。 |
| `failed` | 工具启动后非零退出。 |
| `timed_out` | 到达 timeout，Runtime 已尝试终止整个进程组。 |

## 2. Scope、DNS 与审批保证

实现位于 `src/sec_capsules/core/scope.py`。

执行前，`ScopePolicy.decide()` 会规范化 target，并根据 profile 与 scope 文件产生包含原因的 `ScopeDecision`。真实执行时，它还会解析 hostname 的所有 DNS 地址并逐一检查。

- include / exclude 支持 URL glob、`*.example.com`、host:port 与 CIDR。
- metadata、组播、未指定、保留地址会被拒绝；private/loopback/link-local 只有 `allow_private_ip: true` 时可用。
- `profile.active` 与 `allow_active_scan` 必须兼容。
- profile 声明的速率不能高于 Scope 声明的上限。
- scope 的 `require_approval_for` 或 profile 的 `requires_approval` 命中时，approval 文件必须提供 ID、审批人、允许 action 和目标绑定。

approval 是本地审计记录，表示一次可追踪的人类确认；它不证明目标归属，也不能替代组织的书面授权。

### 重要限制

Scope 当前验证的是初始 target 与 DNS 解析结果，不是内核级或代理级的网络 egress sandbox。会自行发现并访问新 URL 的工具仍须依赖自身的 scope 参数。并且当前字段 `max_requests_per_minute` 与 ProjectDiscovery 常见 `-rl` 的每秒语义尚未统一，v0.1.1 只比较数值；该问题列入 v0.2。

## 3. 证据与上下文隔离保证

- stdout、stderr 与结果 JSON 写入每次 run 独立目录；artifact 有 SHA-256 和相邻元数据文件。
- `ArtifactStore` 仅接受 `artifact://run_id/artifacts/name#Lx` 引用，并限制可读取的行数和字符数，拒绝路径遍历。
- parser 把异构输出转成统一的 services、endpoints、findings、evidence；evidence 指回 artifact 行。
- `ObservationPacket` 在 token 预算内只保留有限代表性结果和计数摘要。
- Recipe 不透传所有 step 的完整 structured payload 或 raw artifact，只保留 step 状态、Observation 和证据引用。

当前尚未实现通用 redaction engine，因此“不默认传给模型”不等于“artifact 已脱敏”。对敏感目标运行时，应将 artifact 存储、访问权限与脱敏策略放在调用环境中控制。

## 4. Recipe 与 MCP 门槛

`core/recipe.py` 仅执行静态 YAML 定义的 DAG：重复 step ID、未知依赖、依赖环会在执行前报错；失败依赖的后续步骤标记 `skipped`。它不是动态 Agent planner。

MCP 额外要求 host 显式设置：

```bash
export SEC_CAPSULES_ALLOW_MCP_EXECUTE=1
```

没有该环境变量，`run_capsule(execute=true)` 与 `run_recipe(execute=true)` 会被拒绝。即便设置了，仍然继续经过 Scope、approval 与预检，不存在“开关绕过安全检查”。

## 5. 已验证的本机 E2E

`scripts/e2e-local.sh` 只针对仓库自带的 Juice Shop compose 环境：

1. Docker Compose 启动靶场，端口仅绑定 `127.0.0.1:3000`。
2. `curl` 等待本机服务就绪。
3. `sec-capsules doctor --require` 验证 `httpx`、`katana`、`nuclei`。
4. 运行 `web-recon-local-lab`：`httpx safe -> katana safe -> nuclei local_lab`。
5. 输出写到 `runs-e2e/`，无论成功失败都由 shell `trap` 清理容器。

GitHub Actions 的 `.github/workflows/e2e-local.yml` 只能手动触发，运行在腾讯自托管 runner 上，并上传 `local-e2e-runs` artifact。它不随普通 push 自动触发，也不会扫描公网目标。
