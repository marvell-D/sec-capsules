# sec-capsules v0.2.0a1 中文开发者手册

## 1. 先理解本版到底证明什么

`sec-capsules` 是 AI Agent 与安全工具之间的一层受控调用模块，不是 Agent 框架，也不是自主渗透系统。上层 Agent 负责理解任务、选择工具和给出语义参数；本项目负责把一次工具调用变成可验证、可授权、可执行、可审计、可压缩的确定性流程。

v0.2.0a1 只证明一个新命题：v0.1.2 为 HTTP JSONL 工具形成的契约，可以在不改写 Runner 主流程的前提下承载 Nmap 的主机、端口、服务和 XML 输出。

本版已经完成：

1. `nmap` Capsule、两个受限 profile、XML fixture 与 parser。
2. `RateLimit` 带单位抽象，同时表达 HTTP RPS 与 Nmap PPS。
3. `assets/services/endpoints/findings/evidence` 五类统一输出。
4. 更强的全 Capsule conformance test。
5. 本机 crAPI Nmap 真实 E2E，且只允许手动触发。
6. Core 外部的 SiliconFlow 两阶段模型评测 adapter。

本版没有完成：FFUF、高基数结果压缩、SARIF、完整对象级 Schema 验证。因此版本是 `0.2.0a1`，而不是稳定的 `0.2.0`。

## 2. 一次调用的完整数据流

```text
Agent / Eval consumer
  |
  | 1. search_capsules: 只看 brief 卡
  | 2. get_capsule usage: 只看选中工具的参数知识
  | 3. 提交 capsule_id / target / profile / arguments
  v
resolve_arguments
  |-- input_schema: 参数的绝对边界
  |-- profile.defaults: 安全默认值
  `-- profile.allowed_arguments: 当前预设允许调整的子集
  v
build_command_plan
  |-- 编译 argv，不经过 shell
  `-- 提取 RateLimit(argument, value, unit)
  v
ScopePolicy.decide
  |-- target / DNS / 私网 / active action
  |-- max_rates[unit]
  `-- approval 是否绑定 action 与 target
  v
inspect_tool -> run_command 或 fixture replay
  v
ArtifactStore 保存 raw output
  v
Capsule parser -> 五类 structured objects
  v
ObservationPacket 按 token budget 压缩
  v
Agent 只看到 observation；需要时再按 artifact ref 取证据片段
```

最重要的职责分离是：模型提交的是 `ports=[5500,8025]`，不是 `-p 5500,8025`；`-sT`、`-sV`、`-oX -`、超时和重试策略都由 Capsule 作者锁定。

## 3. 项目目录和每层责任

```text
sec-capsules/
├── src/sec_capsules/
│   ├── core/                    通用 Runtime，不 import 具体扫描器
│   ├── capsules/                每个工具的声明、parser、fixture、资源
│   │   └── nmap/
│   │       ├── capsule.yml
│   │       ├── parser.py
│   │       └── fixtures/sample.xml
│   ├── evals/                   测试消费者，不属于 Runtime 执行链
│   │   └── providers/siliconflow.py
│   ├── interfaces/              CLI/MCP 等薄适配层
│   ├── recipes/                 确定性 DAG，不做自主规划
│   └── schemas/                 公共 JSON 数据契约草案
├── evals/scenarios/             Provider-neutral 评测题目
├── evals/candidates/reference/  确定性参考答案
├── examples/crapi-local/        本机靶场 Scope 与 approval
├── scripts/                     CI 和手动 E2E 入口
├── tests/                       单元、契约和 provider mock 测试
└── .github/workflows/           GitHub Actions 编排
```

这种结构应用了 Ports and Adapters 思想：Core 是稳定业务规则；CLI、MCP、SiliconFlow 和外部二进制是 adapter。新增模型供应商不应改变 Runner，新增 Capsule 也不应在 Runner 里增加 `if tool == "nmap"`。

## 4. Core 数据结构

### 4.1 `core/models.py`

`Capsule` 保存 `id`、文件路径、根目录和 YAML 原始字典。`profile(name)` 是统一的 profile 查询入口；找不到时抛错，避免调用者静默回退到更危险的预设。

`RateLimit` 是冻结 dataclass：

```python
RateLimit(
    argument="packets_per_second",
    value=20,
    unit="packets_per_second",
)
```

三个字段分别说明：这个值来自哪个语义参数、数值是多少、Scope 应查哪个单位。冻结对象保证 CommandPlan 生成后不会被后续步骤偷偷修改。

`CommandPlan` 是“可执行计划”，不是“执行许可”。它包含 Capsule/profile、target、生效 arguments、每个值的来源、锁定后的 argv、action/active/approval 要求和 `RateLimit`。兼容属性 `requests_per_second` 只在单位确实为 RPS 时返回数值，避免把 PPS 错当成 HTTP RPS。

`RunResult` 是一次调用的审计清单，记录状态、计划、scope decision、工具健康、artifact、结构化输出、observation、错误与时间。其 `rate_limit` 是通用字典；旧 `requests_per_second` 仅作为兼容输出存在。

### 4.2 `core/arguments.py`

`validate_input_schema(capsule)` 验证项目支持的 JSON Schema 子集。它还检查 `x-rate-limit-unit` 只能放在数值类型上、必须是非空字符串、一个 Capsule 最多只能声明一个速率参数。限制为一个是当前版本的主动取舍：Runner 还没有定义“一个工具同时有请求速率和并发数时如何联合授权”。

`resolve_arguments(capsule, profile, provided)` 的顺序是：

1. 验证 Schema 和 profile 配置本身。
2. 拒绝未知字段、当前 profile 未开放字段和不可由 Agent 设置的字段。
3. 深拷贝 profile 默认值，再用 Agent 值覆盖。
4. 对最终值检查类型、范围、枚举、数组长度和唯一性。
5. 返回 `ResolvedArguments(values, sources)`。

越界值不会被自动截断。例如请求 PPS 500 而 Schema 上限是 200，会直接报错。静默改成 200 会让 Agent 误以为自己的计划被原样执行，破坏审计语义。

`resolve_rate_limit(capsule, values)` 查找带 `x-rate-limit-unit` 的参数并构造 `RateLimit`。因此 Planner 不需要知道 Nmap 或 Katana 的字段名。

`template_value(value)` 把经过验证的标量变成字符串，把数组用逗号连接。Nmap 的 `[80, 443]` 因而编译为一个 argv token `80,443`；整个过程不拼 shell 命令。

### 4.3 `core/planner.py`

`build_command_plan()` 组合 `resolve_arguments()` 与 Capsule command template。模板只允许 `$target`、`$capsule_root` 和已经验证的 `$argument`。使用严格替换，模板残留未知变量会立刻失败。

计划里的 command 是 `list[str]`，最终交给 `subprocess.Popen`，不会进入 `/bin/sh -c`。这同时减少命令注入面，也让数组边界和测试断言更清晰。

### 4.4 `core/scope.py`

`ScopePolicy.__init__()` 读取 include/exclude、私网策略、active scan 策略、审批 action 和 `max_rates`。推荐配置是：

```yaml
scope:
  max_rates:
    requests_per_second: 10
    packets_per_second: 30
```

`_set_compatible_rate()` 用于迁移旧 RPS/RPM 字段。新旧字段给出同单位但数值冲突时拒绝启动，而不是猜哪个优先。

`normalize_rate_limit()` 把新 `RateLimit` 和旧函数参数统一成一种内部表示。未知单位没有对应 Scope 上限时 fail closed。

`decide()` 依次验证目标格式、include/exclude、DNS 解析结果、私网/metadata 地址、active 许可、速率和 approval，返回 `ScopeDecision`。`raise_if_denied()` 把所有拒绝理由一起抛出，让上层能一次修正完整计划。

Scope 只约束初始 target 与声明速率，不是内核级网络沙箱。会自行派生 URL 的工具仍需要工具级 scope 参数和隔离网络。

### 4.5 `core/runner.py`

`CapsuleRunner.plan()` 只生成计划，适合 Agent 在执行前检查。

`doctor()` 用 Capsule 的 `runtime.binary/version_command` 检查工具是否存在并记录版本。

`run()` 是主编排函数：生成 run ID、计划、Scope 决策、工具预检、真实执行或 fixture 回放、写 artifact、动态调用 parser、构建 ObservationPacket、写 `run.json`。任何终态都尽量落审计记录；缺工具成为 `preflight_failed`，而不是一段无法归因的 traceback。

真实执行与 fixture 回放共用计划、parser、artifact 和 observation 后半段，所以 fixture 能覆盖绝大多数确定性契约；它不能替代二进制参数兼容性和真实网络行为 E2E。

## 5. Nmap Capsule 的安全设计

### 5.1 `capsules/nmap/capsule.yml`

Agent 只能设置两个参数：

| 参数 | 范围 | 目的 |
|---|---:|---|
| `ports` | 1 至 64 个唯一整数，每个 1..65535 | 避免模糊范围和无限扩张。 |
| `packets_per_second` | 1..200 | 声明本次 PPS，并交给 Scope 二次限制。 |

`safe` profile 固定使用 TCP connect、跳过主机发现、禁用 DNS、限制重试、60 秒 host timeout、XML stdout；它不做版本探测。

`service` profile 额外锁定 `-sV --version-light`，host timeout 为 90 秒。两个 profile 都是 active，都要求 approval。它们都不开放 UDP、SYN 特权扫描、CIDR/range、任意额外参数或 NSE script。

Schema 上限 200 是 Capsule 能力边界；crAPI Scope 上限 30 是具体环境授权边界。Agent 请求 100 时会通过 Schema，但在该环境被 Scope 拒绝。这体现“工具能做什么”和“本次允许做什么”是两套不同规则。

### 5.2 `capsules/nmap/parser.py`

`parse()` 使用标准库 `XMLPullParser` 增量读取 XML。它不依赖 Nmap Python SDK，也不访问网络。处理步骤是：

1. `_completed_hosts()` 只收集已经完整闭合的 `<host>`。
2. 跳过状态不是 `up` 的 host。
3. `_primary_address()` 优先选择 IPv4，再选其他地址。
4. 生成 asset，保留 hostname 与 evidence ref。
5. 只把 `state=open` 的端口映射成 service。
6. 提取 transport、service name、product、version、extra info、tunnel、confidence、CPE。
7. `_service_url()` 只对可识别 HTTP/HTTPS 服务生成 URL。

`_element_lines()` 和 `_address_line()` 在原始 XML 上建立近似行号，使对象能引用 `artifact://<run>/artifacts/nmap.xml#Lx`。ElementTree 不保留精确源码位置，因此这里的行号是证据导航，不应被宣传为 XML AST 的精确 source map。

截断策略是“保留已完成 host，丢弃未闭合尾部”。这对工具超时或 stdout 被上限截断时很重要：前面已完成的资产仍有价值，但 parser 不会伪造半个端口对象。

## 6. 五类结构化输出与上下文管理

所有 parser 必须返回：

```json
{
  "assets": [],
  "services": [],
  "endpoints": [],
  "findings": [],
  "evidence": []
}
```

Nmap 主要产生 assets/services，Katana 主要产生 endpoints，Nuclei 主要产生 findings。统一形状让 Runner、Recipe 和上层 Agent 不必根据扫描器解析不同顶层协议。

`build_observation_packet()` 只把适合模型决策的摘要放入上下文，记录 `new_assets/new_services/new_endpoints/new_findings` 与计数。预算不足时先减少对象数量，再删除空的可选数组。raw XML/JSONL 始终留在 ArtifactStore；Agent 需要核查时按 evidence ref 请求有限片段。`hidden_from_model.secrets_redacted` 当前必须为 `false`：隐藏 raw output 不等于已经脱敏，只有未来接入真实 redaction engine 后才能改变该声明。

这正是项目对“注意力牵引”的处理：工具 banner、重复端口状态、无关 XML 节点和海量原始行不会自动进入下一轮模型上下文。

## 7. SiliconFlow Eval adapter 为什么不属于 Core

`evals/providers/siliconflow.py` 是一个真实消费者，用来回答“模型看渐进式工具知识后，能否选对 Capsule 并生成合法语义参数”。如果把它放进 Core，项目就会开始依赖供应商 SDK、模型选择、重试策略，边界会滑向 Agent 框架。

`SiliconFlowClient` 的函数责任：

| 函数 | 责任 |
|---|---|
| `__init__()` | 只从构造参数或 `SILICONFLOW_API_KEY` 取密钥；生产 CLI 没有 key 参数。 |
| `list_chat_models()` | 调用实时模型列表，返回当前账户实际可用的 chat model ID。 |
| `choose_model()` | 显式模型必须在实时列表内；默认按维护的偏好顺序选择。 |
| `chat_json()` | temperature 0、JSON mode，解析候选并记录 latency/usage/trace ID。 |
| `_request()` | 统一 Authorization、JSON 编码和 transport 注入点。 |
| `_parse_json_object()` | 接受纯 JSON，也容忍模型加代码围栏或少量包裹文字。 |
| `_urlopen_json()` | 标准库 HTTP 实现，把 API/网络错误转成明确异常。 |

`evaluate_scenario()` 分两次调用模型：第一次只给所有候选的 brief 卡，让模型返回 `capsule_id`；第二次只给选中 Capsule 的 usage 卡，让模型返回 target/profile/arguments。最终仍调用本地 `grade_candidate()`，检查预期工具、参数约束、禁止字段、最小权限和 CommandPlan 可编译性。

这样测试的不是“模型会不会背 Nmap flag”，而是“知识渐进披露和语义参数协议能否被模型正确消费”。

密钥规则很简单：真实密钥只能存在于当前进程环境或 CI secret，绝不能进入命令参数、fixture、candidate、run artifact、测试快照或 Git 历史。已公开粘贴过的密钥必须先吊销再测试。

## 8. 测试分层：模块自己不能成为 Agent，仍然可以测性能

本项目采用测试消费者，而不是内置 Agent loop。

### 8.1 单元与契约测试

`tests/test_arguments.py` 验证 Nmap 端口数组、范围、唯一性、模板编译以及无法提交 raw argv。

`tests/test_scope.py` 验证 PPS/RPS 独立授权、未知单位 fail closed、旧 RPM 换算和新旧配置冲突。

`tests/test_parsers.py` 验证 fixture 映射和截断 XML salvage。

`tests/test_capsule_conformance.py` 遍历所有内置 Capsule，验证 metadata、Schema、每个 profile 的默认 CommandPlan、runtime、primary artifact、fixture、parser import 和五类返回集合。这是“新增 YAML 不会到运行时才坏”的 CI 门槛。

`tests/test_siliconflow.py` 使用注入的 fake transport，不访问互联网、不消耗 token。它验证两阶段披露、模型选择、Authorization header、JSON 候选、本地评分和无 CLI 密钥入口。

### 8.2 Fixture replay

fixture 回放测试 parser、artifact、ObservationPacket 和 run manifest，速度快且确定，可在每次 push 执行。它不证明本机安装的 Nmap 版本仍接受这些 argv。

### 8.3 真实本机 E2E

`scripts/e2e-crapi-nmap.sh` 执行：

1. 检出经过审阅的 crAPI compose 源码提交。
2. 强制 `LISTEN_IP=127.0.0.1`，避免靶场暴露到公网网卡。
3. 用独立 Compose project name 启动并等待健康。
4. 对 `127.0.0.1` 的 5500/8025/8443/8888 执行审批后的 Nmap service profile。
5. 断言 structured services 包含四个预期端口。
6. trap 无论成功失败都执行 `down --volumes --remove-orphans`。

它验证 binary 参数、Scope/approval、真实 XML、parser 和 artifact 闭环。它不会自动运行在 push CI，因为第三方镜像下载慢、资源消耗大，也没有必要让每次文档提交都启动易受攻击靶场。

### 8.4 模型能力评测

`evals/scenarios/nmap-crapi-services.yml` 与参考 candidate 让本地 Harness 确定性验证评分逻辑。SiliconFlow live eval 再测模型选择和参数生成。建议记录成功率、两阶段 token、延迟、非法参数率和重试次数；不要只看一次“跑通”。

Planner benchmark 只测本模块生成计划的开销，不设固定毫秒 CI 门槛，因为共享 runner 负载会制造假回归。

## 9. CI/CD 到底如何运行

CI 是持续集成：push/PR 后，在干净环境安装代码并执行确定性检查，尽早发现依赖、打包、测试或契约错误。

CD 在本项目当前阶段不是自动发布到生产，而是把验证后的代码推到 GitHub、保留 Actions 记录和测试 artifact。没有自动把安全工具部署成公网服务。

普通 `.github/workflows/ci.yml` 负责快速、可重复的测试与构建。`scripts/ci.sh` 默认调用 `python3`，也可通过 `PYTHON=/path/to/python` 显式覆盖，避免仍保留 Python 2 的服务器误用旧解释器。`Local E2E` 和 `crAPI Nmap E2E` 都是 `workflow_dispatch`，必须由维护者手动点击运行；两者在腾讯自托管 Linux runner 上执行。

crAPI workflow 的步骤是 checkout、setup Python 3.12、editable install、运行脚本、无论成功失败都上传 `runs-crapi-e2e`。上传的是本次测试的审计产物，不应包含 API key。

常用本地命令：

```bash
PYTHONPATH=src scripts/ci.sh
python -m build
python -m sec_capsules.evals.cli grade \
  --scenario evals/scenarios/nmap-crapi-services.yml \
  --candidate evals/candidates/reference/nmap-crapi-services.json
```

## 10. 修改代码时的思考顺序

新增语义参数时，先问它是 Agent 的决策，还是 Capsule 作者必须锁定的执行不变量。只有前者进入 `input_schema`。

新增工具时，先设计最小风险 profile 和稳定机器输出，再写 parser；不要先把工具全部 flags 暴露出去。一个高质量 Capsule 至少需要 capsule.yml、fixture、parser、参数负向测试和 conformance 通过。

修改 Core 公共数据结构时，检查 CLI、MCP、Recipe、run.json、Schema、所有 parser 和旧兼容属性。Core 改动的测试面应大于单个 Capsule 改动。

优化模型上下文时，应测“模型完成任务所需信息是否足够”和“无关信息是否减少”，而不是单纯追求字节最小。brief 卡负责工具选择，usage 卡负责参数生成，full 卡主要给人类调试。

## 11. v0.2 接下来怎么走

下一主任务是 FFUF Capsule，用高基数输出验证去重、代表性采样和 ObservationPacket 压缩。它应继续使用显式语义参数、受控 wordlist reference、速率 Scope 和 approval，而不是开放任意 CLI。

随后补对象级 Schema conformance、Markdown asset/service 分节和 SARIF finding 原型。只有 Nmap 与 FFUF 都通过 fixture、单元测试和受控 E2E，并且导出与速率语义完成，才能发布稳定 `v0.2.0`。

本项目的衡量标准不是“接了多少扫描器”，而是新增工具后仍能保持：Agent 不接触 argv，调用受 Scope/approval 约束，raw output 不污染上下文，任何结构化结论都能回到 artifact evidence。
