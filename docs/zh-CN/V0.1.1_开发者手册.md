# sec-capsules v0.1.1 开发者手册

本文面向刚接手本仓库、希望理解每个模块职责并能安全修改代码的开发者。它描述的是当前 `0.1.1` 已经实现的行为，不把未完成的设想写成既有能力。

## 1. 先建立正确心智模型

`sec-capsules` 不是 AI 渗透框架，也不负责决定“下一步攻击什么”。它解决的是更窄的一层问题：当上层 Agent、CLI 或 CI 已经决定要使用一个安全工具时，如何让这次工具调用可描述、可约束、可审计、可解析，并避免把整段原始输出直接塞给模型。

一次调用的主数据流如下：

```text
用户 / Agent 请求
    -> 读取 Capsule 工具卡
    -> 生成 argv CommandPlan
    -> ScopePolicy 判定范围、速率和审批
    -> ToolHealth 预检外部二进制
    -> 受控子进程执行 / fixture 回放 / dry-run
    -> 原始 stdout、stderr 写入 Artifact
    -> Capsule 专属 Parser 转成统一对象
    -> ObservationPacket 按 token 预算摘要
    -> RunResult 与 run.json 落盘
```

这个分层的价值在于：上层接口不需要知道 `httpx`、`katana`、`nuclei` 的 JSONL 字段细节；工具本身也不需要知道 Agent 的上下文窗口有多大。两者通过 Capsule、统一对象、artifact ref 和 ObservationPacket 协作。

## 3. 安装与日常命令

### 3.1 本地 Python 环境

```bash
cd /home/ubuntu/git/sec-capsules
python3 -m venv .venv
. .venv/bin/activate
python -m pip install -e .
```

`-e` 是 editable install。Python 包仍从当前工作目录的 `src/` 读取，因此修改源码后通常不需要重新安装。`pyproject.toml` 定义了：

| 配置项 | 含义 |
|---|---|
| `build-system` | 使用 setuptools 按 PEP 517 构建 wheel 与 sdist。 |
| `project.dependencies` | Runtime 只强依赖 `PyYAML`，因为 Capsule、Scope、Recipe 都是 YAML。 |
| `mcp` optional extra | 只有启动 MCP server 时才需要安装。 |
| `dev` optional extra | 声明 pytest，当前主 CI 实际使用标准库 `unittest`。 |
| `project.scripts` | 安装后暴露 `sec-capsules` 命令，入口是 `interfaces.cli:main`。 |
| `package-data` | 将 YAML、fixture、Nuclei local template、recipe、schema 一起打进 wheel，避免安装后找不到配置文件。 |

### 3.2 最常用的命令

```bash
# 列出工具卡
sec-capsules list

# 检查外部二进制是否存在；--require 会在缺工具时返回非零状态
sec-capsules doctor --require

# 不执行，只展示最终 argv 计划
sec-capsules plan nuclei --target https://example.com --profile safe

# 使用仓库 fixture 回放，适合开发 parser，不访问网络
sec-capsules run nuclei \
  --target http://localhost:3000 \
  --scope examples/juice-shop-local/scope.yml \
  --fixture src/sec_capsules/capsules/nuclei/fixtures/sample.jsonl

# 真实执行：只可用于已授权目标，必须显式提供 --execute
sec-capsules run httpx \
  --target http://127.0.0.1:3000 \
  --scope examples/juice-shop-local/scope.yml \
  --execute

# 读取一条受限 evidence，不允许直接读取任意本机路径
sec-capsules artifact get artifact://run_xxx/artifacts/nuclei.jsonl#L1

# 导出某次运行为 Markdown
sec-capsules export run_xxx --format markdown
```

全局 `--runs-dir` 可放在子命令前，也可放在子命令后：

```bash
sec-capsules --runs-dir my-runs run httpx ...
sec-capsules run httpx ... --runs-dir my-runs
```

这是由 `argparse.SUPPRESS` 处理的：子命令没有显式传值时，不会意外覆盖全局参数。对应回归测试在 `tests/test_cli.py`。

## 4. 核心数据结构：先理解“项目传递什么”

所有核心数据类定义在 `src/sec_capsules/core/models.py`，使用 Python `@dataclass`。这比在各模块间传无约束的字典更容易阅读、序列化和测试。

| 数据结构 | 关键字段 | 谁创建 | 为什么存在 |
|---|---|---|---|
| `Capsule` | `id`、`raw`、`root` | `CapsuleRegistry` | 把一个 `capsule.yml` 及其目录包装成对象。`frozen=True` 防止运行时意外改写工具定义。 |
| `CommandPlan` | `command: list[str]`、`action`、`rate_limit` | `build_command_plan` | 将 profile 的模板变量替换为可执行 argv；它只是计划，不代表已获执行授权。 |
| `ArtifactRecord` | `sha256`、`path`、`content_type` | `ArtifactStore` | 原始证据的索引条目，供 `RunResult` 和导出功能引用。 |
| `RunResult` | `status`、`scope_decision`、`tool`、`artifacts`、`structured`、`observation` | `CapsuleRunner.run` | 一次调用的完整审计快照，写入 `run.json`。 |
| `RunContext` | `target`、`scope_file`、`profile` | 尚未接入主流程 | 为后续扩展预留的上下文载体；当前不要误以为它已承担所有状态管理。 |
| `ToolHealth` | `binary`、`available`、`path`、`version` | `inspect_tool` | `doctor` 和真实执行前的二进制预检结果。 |
| `ExecutionOutput` | stdout、stderr、exit code、超时、截断标记 | `run_command` | 把子进程执行结果从 `subprocess` 细节中抽离。 |
| `ScopeDecision` | `allowed`、`reasons`、解析 IP、approval 摘要 | `ScopePolicy.decide` | 不只回答“允许/拒绝”，还保留拒绝原因与审计上下文。 |

### 4.1 三种不同层次的输出

不要混淆下面三类数据：

1. **artifact**：原始 stdout/stderr，内容可能很大、可能包含敏感数据。它存盘，不默认进入模型。
2. **structured**：parser 产生的标准对象字典，固定含 `services`、`endpoints`、`findings`、`evidence` 四个列表。
3. **ObservationPacket**：面向模型的紧凑摘要，例如最多 5 个 finding、8 个 service、12 个 endpoint，受 token 预算裁剪。

`RunResult` 同时包含三者，是给程序和审计用的记录；Agent 的默认输入应是 ObservationPacket，而不是整个 RunResult。

### 4.2 run 目录布局

假设本次 ID 为 `run_abcd1234`，默认目录为：

```text
runs/
└── run_abcd1234/
    ├── run.json                 # 完整审计 manifest
    ├── structured.json          # 标准对象集合
    ├── observation.json         # token 受限摘要
    └── artifacts/
        ├── nuclei.jsonl         # 原始 stdout
        ├── nuclei.jsonl.artifact.json
        ├── stderr.txt           # 有 stderr 时才存在
        └── stderr.txt.artifact.json
```

`run.json` 的 `status` 可能是 `dry_run`、`replayed`、`succeeded`、`failed`、`timed_out` 或 `preflight_failed`。CLI 只有前三种返回 0；因此 CI 不会把超时误判为成功。

## 5. Core Runtime：逐文件代码导读

### 5.1 `paths.py`：统一资源根目录

| 符号 | 作用 |
|---|---|
| `PACKAGE_ROOT` | 当前安装包 `sec_capsules/` 的根目录。 |
| `CAPSULES_ROOT` | 内置 `capsules/` 目录。 |
| `RECIPES_ROOT` | 内置 `recipes/` 目录。 |
| `SCHEMAS_ROOT` | 内置 JSON Schema 目录。 |

这里使用 `Path(__file__).resolve()`，所以无论项目是 editable install 还是 wheel 安装，代码都能按包自身的位置定位随包分发的 YAML。

### 5.2 `registry.py`：配置驱动的工具插件发现

`CapsuleRegistry.load()` 扫描 `*/capsule.yml`，用 `yaml.safe_load` 读取，随后构造 `Capsule`。第一次调用后将字典缓存在 `_capsules`，避免一次命令反复读盘。

| 函数 / 方法 | 输入 | 输出 | 教学重点 |
|---|---|---|---|
| `load()` | 无 | `dict[str, Capsule]` | 文件系统 discovery + lazy cache。 |
| `list()` | 无 | Capsule 列表 | 给 CLI list 与搜索使用。 |
| `get(id)` | Capsule ID | 单个 Capsule | 未找到时展示可选 ID，利于人和 Agent 调试。 |
| `search(query, stage, risk_level)` | 过滤条件 | Capsule 列表 | 用可读字段拼成 haystack；v0.1.1 不引入复杂索引。 |
| `capsule_to_public_dict()` | Capsule、detail level | 面向接口的 dict | **渐进披露**：brief 不给 profile 细节，usage 给适用场景，full 才返回原 YAML。 |

这是一种轻量插件架构：新增工具不需要改注册表，只需新增规范目录。缺点是 YAML 在运行时才校验，v0.2 的 conformance test 将补强这一点。

### 5.3 `planner.py`：把工具卡变成 argv

`build_command_plan(capsule, target, profile_name)` 读取 profile 的 `command` 列表，用 `string.Template` 替换：

| 模板变量 | 来源 |
|---|---|
| `$target` | 调用者传入目标。 |
| `$rate_limit` | profile 中的 `rate_limit`。 |
| `$severity` | profile 的 severity 列表拼为逗号字符串。 |
| `$capsule_root` | 当前 Capsule 目录的绝对路径，用于 local-lab Nuclei template。 |
| `vars` 中的键 | Capsule 自定义变量。 |

最终 `command` 是字符串数组，例如 `['httpx', '-json', '-u', 'http://127.0.0.1:3000']`，不是 shell 字符串。这个选择很重要：Runtime 不使用 `shell=True`，从根源避免由引号、分号、管道等 shell 语法造成的注入路径。

### 5.4 `executor.py`：外部进程的受控边界

这是 v0.1.1 最关键的新增模块。

| 符号 | 行为 | 为什么这样设计 |
|---|---|---|
| `inspect_tool()` | `shutil.which` 定位二进制，执行 `-version`，抽取可读版本行。 | 把“工具没安装”变成可审计的 `preflight_failed`，而不是 Python traceback。 |
| `execution_environment()` | 只传 `HOME`、`PATH`、语言、证书、临时目录等少量环境变量。 | 使用环境变量 allowlist，降低无意把 API key 或 CI secret 传给工具的概率。 |
| `extract_version()` | 删除 ANSI 颜色码，优先找含 version 的行。 | ProjectDiscovery 工具会输出 banner，不能盲取第一行。 |
| `run_command()` | `Popen` 启动进程，分线程读取 stdout/stderr，等待超时。 | 独立读取两条 pipe，防止其中一个 pipe 写满导致死锁。 |
| `_StreamState` | 每个流最多保留 `max_output_bytes`。 | 即使工具疯狂输出，Runtime 也保留截断状态而非无限消耗内存。 |
| `_terminate_process_group()` | 先 `SIGTERM`，3 秒后 `SIGKILL`。 | `start_new_session=True` 让工具及其子进程同属进程组，超时能一起终止。 |

这里使用了几个值得学习的系统编程思想：**能力最小化环境**、**进程组管理**、**背压/输出上限**、**终态结果而非异常泄漏**。但它不是容器沙箱；外部工具仍在 runner 用户权限下运行，因此 Scope 和 runner 隔离仍然重要。

### 5.5 `scope.py`：执行前的策略判定

`ScopePolicy` 从 YAML 加载规则，`decide()` 返回 `ScopeDecision`，而非立刻抛异常。只有调用方显式 `raise_if_denied()` 才会中止，这使 CLI、MCP、审计日志都能拿到同一份原因。

`decide()` 的判定顺序为：

1. `parse_target()` 规范化 scheme、host、端口、路径。
2. 检查 active 调用是否被 `allow_active_scan` 禁止。
3. 比较 profile 请求速率与 scope 上限。
4. 根据 hostname、URL 模式或 CIDR 检查 include/exclude。
5. 对真实执行解析 DNS，并检查所有解析地址。
6. 拒绝 metadata IP、组播、未指定地址、保留地址，以及未显式允许的 private / loopback / link-local 地址。
7. 若动作或 profile 要审批，验证 approval 记录的 id、审批人、动作和目标绑定。

| 函数 / 类 | 作用 |
|---|---|
| `ScopePolicy.from_file()` | 从 scope YAML 创建策略。 |
| `approval_from_file()` | 读取 approval YAML；接受顶层 `approval:` 或直接对象。 |
| `decide()` | 返回允许/拒绝、原因、标准化目标、解析 IP、审批摘要。 |
| `parse_target()` | URL 规范化，消除默认端口与 host 大小写差异。 |
| `resolve_host()` | 使用 `socket.getaddrinfo` 得到所有 IP。 |
| `is_blocked_private_host()` | 对 hostname/IP 应用私网与 metadata 规则。 |
| `match_scope_pattern()` | 支持 URL glob、`*.example.com`、host:port 和 CIDR。 |
| `validate_approval()` | 检查 approval 是否真的覆盖当前动作与当前目标。 |

一个 scope 示例：

```yaml
scope:
  include:
    - https://app.example.com
    - "*.app.example.com"
  exclude:
    - https://admin.app.example.com
  max_requests_per_minute: 60
  allow_private_ip: false
  allow_active_scan: false
  require_approval_for:
    - fuzzing
    - credentialed_scan
```

对应审批记录：

```yaml
approval:
  id: apr_20260710_001
  approved_by: security-operator@example.invalid
  actions: [credentialed_scan]
  targets: [https://app.example.com]
```

**当前限制，必须诚实理解：** ProjectDiscovery 的 `-rl` 在当前工具版本中表示“每秒请求数”，而 Scope 字段名仍是 `max_requests_per_minute`。v0.1.1 目前只做数值上限比较，尚未统一单位。不要把这个字段当作精确的 RPM 节流保证；v0.2 应将它重命名或引入明确单位。另一个限制是 Runtime 验证的是工具的初始 target；爬虫随后访问的 URL 依赖工具自身的 scope 参数。Katana safe profile 已使用 `-depth 2` 和 `-fs fqdn`，但这不是全局网络代理级的出站强制控制。

### 5.6 `artifacts.py`：证据存储与受限读取

| 方法 | 行为 |
|---|---|
| `write_text()` | 写文件、计算 SHA-256、生成 `ArtifactRecord` 和相邻 `.artifact.json`。 |
| `write_json()` | 将对象 JSON 化后复用 `write_text()`。 |
| `resolve_ref()` | 只接受 `artifact://run_id/artifacts/name#L12`，拒绝 `..`、绝对路径、越出 artifacts 目录的路径。 |
| `read_ref()` | 读取显式 artifact 的行片段，受 `max_lines`、`max_chars` 限制。 |
| `artifact_ref()` | parser 生成统一的 evidence 引用。 |

这是“证据引用而非证据复制”的设计。模型先看到 `artifact://run_x/artifacts/nuclei.jsonl#L1`，只有确实需要时再请求有限片段。

注意：`ArtifactRecord.redacted` 当前默认是 `false`，项目还没有通用的 redaction engine。v0.1.1 的主要保护是“不默认把 raw artifact 给模型”，不是“已自动删除所有敏感内容”。通用脱敏属于 v0.3。

### 5.7 `parsers.py` 与三个工具 parser：从异构 JSONL 到统一对象

`parse_capsule_output()` 根据 `capsule.id` 动态 import `sec_capsules.capsules.<id>.parser`，要求该模块暴露：

```python
parse(raw_text: str, *, run_id: str, artifact_name: str) -> dict[str, Any]
```

这种按 ID 动态加载的方式是一个小型 **plugin strategy**：Core 不需要 import 每个工具的解析器；新 Capsule 可以自带 parser。

| 文件 | `parse()` 生成什么 | 关键逻辑 |
|---|---|---|
| `capsules/httpx/parser.py` | `service.v1`、`endpoint.v1`、`evidence.v1` | 从 `url`、`host`、`port`、`status_code`、`title`、技术字段提取 HTTP 服务。 |
| `capsules/katana/parser.py` | `endpoint.v1`、`evidence.v1` | 从 JSONL 请求对象取 URL 和 method，用 `seen: set[str]` 去重。 |
| `capsules/nuclei/parser.py` | `finding.v1`、`evidence.v1` | 从 `info`、`template-id`、`matched-at`、classification 生成 finding。 |

所有 parser 都逐行 `json.loads`，坏行直接跳过；这让工具夹杂 banner 或截断尾行时不至于让整个 run 崩溃。每个对象都带 evidence ref，可以定位回原始第几行。

### 5.8 `observation.py`：token 预算下的上下文管理

`build_observation_packet()` 是项目解决“模型注意力被工具输出淹没”的核心函数。

它先保留：执行状态、计数摘要、最多 5 个 finding、最多 8 个 service、最多 12 个 endpoint、推荐下一步。之后按顺序删除 endpoint、service、finding，直到 `estimate_tokens()` 不超过预算。`estimate_tokens()` 目前是 `len(repr(value)) // 4` 的近似值，优点是无 tokenizer 依赖，缺点是并非模型真实 token 数。

| 函数 | 责任 |
|---|---|
| `build_observation_packet()` | 建包、逐步裁剪、写预算元数据。 |
| `summarize_counts()` | 生成面向人和模型的计数摘要。 |
| `recommended_next_actions()` | 根据 findings / endpoints 给出下一步标签，不执行决策。 |

这里体现的是 **信息分级与有损摘要**：完整性留给 artifact，模型上下文优先保留“下一步是否需要行动”的信号。

### 5.9 `runner.py`：一次 Capsule run 的总编排器

`CapsuleRunner` 是所有 Runtime 部件汇合的位置。

| 方法 | 行为 |
|---|---|
| `__init__()` | 接收可替换的 `CapsuleRegistry` 与 runs dir；这是一种依赖注入，测试可传临时 registry。 |
| `plan()` | 只返回 CommandPlan，不执行 Scope 或子进程。 |
| `doctor()` | 对一个或所有 Capsule 调用 ToolHealth 预检。 |
| `run()` | 完整执行生命周期。 |
| `_write_result()` | 把 RunResult、structured、observation 分别持久化。 |

`run()` 的关键分支：

| 场景 | 是否启动外部工具 | `status` |
|---|---:|---|
| 既无 `--execute` 也无 fixture | 否 | `dry_run` |
| 给了 fixture | 否 | `replayed` |
| 外部二进制不存在 | 否 | `preflight_failed` |
| 子进程超时 | 是，随后终止 | `timed_out` |
| 非零退出 | 是 | `failed` |
| 正常退出 | 是 | `succeeded` |

即使失败或超时，只要已有 stdout/stderr，Runner 仍会写 artifact 和 manifest。这样审计与调试不会因异常丢证据。

### 5.10 `recipe.py`：确定性流程，而非 Agent planner

Recipe YAML 只有静态 step 和 `depends_on`。`order_steps()` 通过拓扑排序保证依赖先运行：重复 step id、未知依赖和环都会在执行前报错。

`run_recipe()` 不把每次 run 的完整 `structured` 继续堆进 recipe 返回值，而只保存 step id、run id、状态、Observation、artifact ref。若依赖失败，后续 step 标记为 `skipped`。

| 函数 | 作用 |
|---|---|
| `load_recipe()` | 支持内置 recipe ID 或 YAML 路径。 |
| `order_steps()` | 稳定拓扑排序与 DAG 校验。 |
| `run_recipe()` | 顺序执行准备好的 steps，处理失败依赖，写 recipe_run。 |
| `build_recipe_observation()` | 构建 recipe 级摘要，继续隔离 raw artifact。 |

这是 **显式工作流 + 失败短路** 模式。它可重复、可测试，也刻意不根据扫描结果自动决定“下一步攻击”。

### 5.11 `exporters.py`：面向人的报告出口

`export_markdown(run_dir)` 读取 `run.json`，生成包含执行时间、命令、Observation、top finding 与 artifact SHA-256 的 Markdown。它不复制 raw artifact 内容，仍维持证据分层。

## 6. 接口层：CLI 与 MCP 为什么不写业务逻辑

### 6.1 `interfaces/cli.py`

`main()` 解析参数并统一捕获异常，`build_parser()` 注册命令，`cmd_*` 函数只负责参数适配、调用 Core、打印 JSON、返回退出码。

| CLI 命令 | 对应函数 | Core 入口 |
|---|---|---|
| `list` | `cmd_list` | `CapsuleRegistry.list` |
| `describe` | `cmd_describe` | `capsule_to_public_dict` |
| `search` | `cmd_search` | `CapsuleRegistry.search` |
| `doctor` | `cmd_doctor` | `CapsuleRunner.doctor` |
| `plan` | `cmd_plan` | `CapsuleRunner.plan` |
| `run` | `cmd_run` | `CapsuleRunner.run` |
| `recipe run` | `cmd_recipe_run` | `core.recipe.run_recipe` |
| `observe` | `cmd_observe` | 读取 observation 文件 |
| `artifact get` | `cmd_artifact_get` | `ArtifactStore.read_ref` |
| `export` | `cmd_export` | `export_markdown` |

这是一种 **thin adapter** 思想：CLI 不自己实现 Scope、parser 或 artifact 规则，所以未来改成 HTTP API 也不会复制一份安全逻辑。

### 6.2 `interfaces/mcp_server.py`

MCP adapter 暴露的不是 `nuclei_scan`、`ffuf_fuzz` 这类工具专属入口，而是 7 个元工具：

| MCP 函数 | Agent 可以做什么 |
|---|---|
| `search_capsules` | 按关键词、阶段、风险发现能力。 |
| `get_capsule` | 逐级读取某工具卡。 |
| `run_capsule` | 执行 / 回放一次 Capsule。 |
| `run_recipe` | 执行确定性流程。 |
| `get_observation` | 读取某次摘要。 |
| `get_artifact` | 有限地读取明确 artifact ref。 |
| `export_run` | 取得 Markdown 报告。 |

两个私有辅助函数构成额外边界：

- `_require_mcp_execution_enabled()`：只有服务器进程显式设置 `SEC_CAPSULES_ALLOW_MCP_EXECUTE=1`，MCP 调用的 `execute=true` 才被接受。
- `_run_dir()`：run ID 必须是单个路径组件，避免 `../` 逃出 `SEC_CAPSULES_RUNS_DIR`。

启动 MCP 前先安装 optional dependency：

```bash
python -m pip install -e '.[mcp]'
python -m sec_capsules.interfaces.mcp_server
```

默认不要设置 `SEC_CAPSULES_ALLOW_MCP_EXECUTE`。开启它只说明 MCP host 允许真实执行；Scope、approval 和外部工具预检仍然会继续执行。

## 7. Capsule、Schema、Recipe 与示例文件

### 7.1 每个 Capsule 目录的固定组成

```text
capsules/<tool-id>/
├── capsule.yml              # 工具卡：知识、profile、命令、输出声明
├── parser.py                # 将该工具原始输出映射为统一对象
├── fixtures/sample.jsonl    # 可复现的 golden fixture
└── templates/               # 仅在工具确实需要随包模板时存在
```

当前三个工具卡：

| Capsule | profile | 作用 | 重要安全参数 |
|---|---|---|---|
| `httpx` | `safe` | HTTP 存活探测与标题/状态码收集。 | `-rl 20`，JSONL 输出。 |
| `katana` | `safe` | 端点收集。 | `-depth 2`、`-fs fqdn`、`-rate-limit 10`。 |
| `nuclei` | `safe` | 中高危模板基线扫描。 | `-severity medium,high,critical`、`-rl 5`。 |
| `nuclei` | `local_lab` | 仅供本机 Juice Shop E2E 的单模板验证。 | 显式 `-t` + `-it`，防止跑默认全模板集。 |

`local_lab` 的模板文件是 `capsules/nuclei/templates/local-juice-shop.yaml`。Nuclei 默认 ignore 规则可能排除简单指纹模板，因此该 profile 同时传 `-t` 和 `-it`：前者限定仅一个模板，后者允许该显式模板绕过 ignore。这个细节来自真实 E2E 验证，不是猜测。

### 7.2 Schema 文件

`schemas/` 中的文件是 JSON Schema 草案，当前主要承担命名与形状约束的文档角色：

| Schema | 表示的对象 |
|---|---|
| `asset.v1.json` | 资产对象预留格式。 |
| `service.v1.json` | 网络/HTTP 服务。 |
| `endpoint.v1.json` | 可访问 URL 与方法。 |
| `finding.v1.json` | 安全 finding 候选。 |
| `evidence.v1.json` | finding / endpoint 回溯原始证据的引用。 |
| `artifact.v1.json` | 原始输出文件及 SHA-256。 |
| `run.v1.json` | 一次 Runtime 执行 manifest。 |
| `observation_packet.v1.json` | 面向模型的紧凑摘要。 |

当前 Runtime 尚未在每次写入时调用 JSON Schema validator。v0.2 的 conformance test 应补上“Capsule metadata、parser 返回值和 schema 一致”的自动校验。

### 7.3 Recipe 与本地靶场

| 文件 | 用途 |
|---|---|
| `recipes/web-recon-safe.yml` | 已授权 Web 目标的三步通用纵切。 |
| `recipes/web-recon-local-lab.yml` | CI/E2E 专用，最后一步使用 Nuclei `local_lab`。 |
| `examples/juice-shop-local/compose.yml` | 启动 Juice Shop，并仅将 3000 绑定到 `127.0.0.1`。 |
| `examples/juice-shop-local/scope.yml` | 明确允许本机 localhost / 127.0.0.1:3000。 |
| `examples/approval.example.yml` | 可复制后填写的审批记录结构。 |

## 8. 测试是怎样覆盖设计的

`scripts/ci.sh` 只做两件事：`compileall src tests` 与 `unittest discover -s tests -v`。当前 26 项测试并非为了追求数字，而是围绕风险点建立回归保护。

| 测试文件 | 守护的契约 |
|---|---|
| `test_registry.py` | 能发现内置 Capsule；usage 视图不会泄露 full command。 |
| `test_parsers.py` | 三个 JSONL fixture 可被解析，Katana 去重，Nuclei evidence 指向正确行。 |
| `test_observation.py` | 极小 token budget 下仍不超预算。 |
| `test_scope.py` | in-scope、out-of-scope、metadata IP、DNS 到私网、速率与 approval。 |
| `test_executor.py` | stdout/stderr 截断、超时终态、banner 版本提取。 |
| `test_artifacts.py` | artifact 片段读取与路径遍历拒绝。 |
| `test_runner.py` | fixture run 落盘、plan 内容、缺工具变成 preflight 状态。 |
| `test_recipe.py` | 依赖排序、环检测、recipe 不透传 structured payload。 |
| `test_cli.py` | 全局与子命令 `--runs-dir` 不互相覆盖。 |
| `test_mcp_server.py` | MCP artifact/observation/export 的 runs-dir 隔离，以及默认禁止 live execute。 |

修改任何 Core 行为前，先问自己“应新增或调整哪一个契约测试”。安全工具项目最危险的回归通常不是语法错误，而是某条拒绝路径、超时路径或上下文隔离路径悄悄失效。

## 9. CI/CD：概念、文件、真实执行过程

### 9.1 CI 和 CD 分别是什么

**CI（Continuous Integration，持续集成）** 指每次 push 或 Pull Request 自动把大家的改动整合后测试。它回答：“这次提交还能构建和通过测试吗？”

**CD** 有两个常见含义：

1. **Continuous Delivery，持续交付**：自动生成可交付物，但由人决定何时发布。
2. **Continuous Deployment，持续部署**：通过测试后自动发布到生产环境。

本仓库当前实现的是 **CI + 持续交付**，不是自动部署到 PyPI。安全 Runtime 在 v0.1.1 尚不应自动发布到公共包索引；GitHub Actions artifact 就是经过验证、可下载的 wheel/sdist 交付物。

### 9.2 常规 CI：`.github/workflows/ci.yml`

触发条件：任何 `push` 与 `pull_request`。

```text
test job
  1. checkout 源码
  2. actions/setup-python 安装 Python 3.12
  3. pip install -e .
  4. scripts/ci.sh：compileall + 26 unittest
       |
       v
package job（needs: test）
  1. 重新 checkout 与设置 Python
  2. pip install build
  3. python -m build -> dist/*.whl + dist/*.tar.gz
  4. upload-artifact -> sec-capsules-dist
```

`needs: test` 是 job dependency，表示测试失败时 package 不会开始。`actions/upload-artifact@v4` 上传的文件可以在 Actions run 页面下载，不会自动发布。

### 9.3 手动 Local E2E：`.github/workflows/e2e-local.yml`

该 workflow 只使用 `workflow_dispatch`，所以不会因为普通 push 自动运行。原因是它真的会在 runner 上启动 Docker 靶场和真实工具进程。

执行顺序：

```text
手动点击 Run workflow / gh workflow run
  -> checkout
  -> Python 3.12 + editable install
  -> scripts/e2e-local.sh
       -> docker compose 启动本机 Juice Shop
       -> curl 等待 127.0.0.1:3000 就绪
       -> sec-capsules doctor --require
       -> recipe run web-recon-local-lab --execute
       -> trap 清理容器和网络
  -> 无论成功失败都上传 runs-e2e/ 为 local-e2e-runs artifact
```

`if: always()` 很重要：即使某个工具失败，run manifest 和 stderr artifact 仍会被上传，便于诊断。

### 9.4 自托管 Runner 是什么

GitHub-hosted runner 是 GitHub 临时提供的虚拟机；self-hosted runner 是你自己的服务器进程向 GitHub 报到后领取 job。本项目使用腾讯服务器上的 `ubuntu` 用户、Linux 标签和 systemd service。

优点：能使用服务器上已安装的 Docker、`httpx`、`katana`、`nuclei`，也能运行只绑定 loopback 的 Juice Shop。代价：你必须自己维护系统补丁、磁盘、网络、runner 升级和 secrets 边界。

当前服务器上外部工具安装到 `/usr/local/bin`，Runner service 以 `ubuntu` 运行。不要把 runner registration token、GitHub PAT、`.credentials` 或 `.service` 文件加入仓库。

## 10. 当前版本的已知边界

理解边界比罗列功能更重要：

- 没有 Agent 自主攻击规划、漏洞利用、凭据管理或多 Agent 协作。
- 没有通用 redaction engine；raw artifact 默认不进模型，但不等于 artifact 内容已自动脱敏。
- 没有签名审批、审批过期、组织授权验证；approval 记录是审计确认，不是法律授权证明。
- Scope 不是透明代理，不能拦截外部工具内部的每一个后续请求。
- `max_requests_per_minute` 字段与 ProjectDiscovery `-rl` 的单位尚未统一，不能把它视作精确 RPM 控制。
- `safe` 是受限 profile 名称，不等于目标天然安全。尤其 Nuclei 通用 safe profile 可能加载较大模板集；E2E 使用单独 `local_lab` profile 才保证快速、确定。
- schema 已随包提供，但还未在所有写入点执行严格 validator。

这些限制不是失败，而是 v0.2 / v0.3 应有的明确工作项。写文档时把限制讲清楚，反而会让开源项目更可信。

## 11. v0.2：为什么是 nmap + ffuf，以及具体要做什么

v0.2 的目标不是“多两个命令”，而是证明 Core Runtime 可以处理输出形态、风险模型和上下文噪声完全不同的工具。

### 11.1 `nmap` Capsule：证明非 HTTP 服务模型

预期工作：

1. 建立 `capsules/nmap/capsule.yml`，定义经过授权的低风险 profile、二进制预检和 artifact 名称。
2. 选择并固定 XML 输出，新增 XML parser，而不是依赖人读文本表格。
3. 把 host、port、transport、service、product、version、状态映射到扩展后的 `service.v1`。
4. 增加真实 XML fixture、边界 parser 测试和 scope/CIDR 测试。
5. 明确端口扫描的 active 风险与 approval 语义。

它考验的是：Runtime 是否只适配 JSONL Web 工具，还是能够容纳不同序列化格式与服务发现语义。

### 11.2 `ffuf` Capsule：证明高噪声结果治理

预期工作：

1. 定义 fuzzing action，默认要求 Scope 中的 `fuzzing` approval。
2. 使用 JSON 输出并保存完整 artifact。
3. 设计 parser，把 URL、status、length、words、lines、payload 等转为候选 endpoint / finding。
4. 设计稳定去重键，例如规范化 URL + method + status + 响应特征，而不是只按原始行去重。
5. 引入基线过滤、相似响应聚类、阈值化摘要，避免把成百上千条 fuzz 结果推给模型。

它考验的是：ObservationPacket 在“结果很多但信息密度低”时是否仍然有效。

### 11.3 v0.2 的共同工程任务

- Capsule conformance test：检查必填字段、profile command、runtime metadata、fixture、parser 是否齐全。
- 统一 schema：补全 service / endpoint / finding 的字段与版本兼容策略。
- 改进 Markdown 导出，并新增 SARIF 导出，方便 CI 安全结果接入其他平台。
- 为 nmap、ffuf 设计明确的 fixture 回放与真实本地 E2E，不将第三方目标放入测试。
- 解决速率单位问题，并评估更严格的网络出站约束方式。

### 11.4 v0.2 完成标准

到 v0.2 结束时，应该能诚实地展示：同一个 `CapsuleRunner` 不修改核心流程，就能处理 ProjectDiscovery JSONL、Nmap XML、FFUF 高噪声 JSON；每个 Capsule 都有工具卡、fixture、parser、conformance test、scope-safe profile、artifact ref 和 Observation 策略。

## 12. 给下一位开发者的修改路线

新增一个 Capsule 时，按以下顺序工作：

1. 先写 `capsule.yml`：明确工具适合什么、不适合什么、风险级别、profile、输出和 artifact。
2. 收集一个真实但脱敏且授权的 fixture，先写 parser 测试。
3. 写 parser，并确保坏行、空输出、重复结果不会崩溃。
4. 用 `sec-capsules plan` 审核 argv，而不是直接把命令塞进 Runner。
5. 决定 Scope action 与 approval 需求。
6. 为真实执行写受控本地 E2E，不碰未授权目标。
7. 补 README、Capsule Spec 与本手册中的索引。

如果发现自己开始写“根据 finding 自动选择下一个攻击工具”“自动尝试 exploit”“保存长期攻击记忆”，应当停下来检查项目边界。这些不是 sec-capsules 当前要解决的问题。
