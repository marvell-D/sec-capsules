# sec-capsules v0.1.2 开发者手册

本文解释 v0.1.2 的 Agent 参数协议和测试体系。阅读目标不是“会运行一条命令”，而是理解为什么每一层存在、数据怎样经过每个函数、修改时应守住什么契约。

基础的子进程、Scope、artifact、parser、ObservationPacket 和 CI/CD 细节仍见[v0.1.1 开发者手册](V0.1.1_开发者手册.md)。本文不复制那 600 多行基础说明，而是在它之上讲清本次变化。

## 1. v0.1.1 的问题到底是什么

v0.1.1 的 Agent 可以做两件事：选择 Capsule，选择 `safe`/`local_lab` profile。真正的工具参数却固定在 YAML 中：Katana 永远 depth 2，Nuclei 永远 medium/high/critical。

这对验证真实执行门槛很合适，但对通用插件不够。Agent 即使读到完整 YAML，也没有 `arguments` 通道把“只爬一层、每秒三个请求”传给 Planner。

v0.1.2 把职责重新分开：

```text
Agent 负责：根据任务选择语义参数
Capsule 负责：描述参数知识、默认值、允许范围和 argv 模板
Runtime 负责：校验、编译、Scope/approval、执行和审计
外部工具负责：真正的扫描/爬取行为
```

Agent 不负责拼 `katana -depth 1 ...`。如果把 argv 直接交给模型，模型就可能关闭 JSON 输出、改变 scope flag、启用未经批准的模板或传入项目根本不理解的参数。

## 2. 一次调用的完整数据流

```text
1. search_capsules()
   返回多张 brief 工具卡

2. get_capsule("katana", "usage")
   返回 input_schema + profile defaults + allowed_arguments

3. Agent 生成 InvocationRequest
   {capsule_id, target, profile, arguments}

4. resolve_arguments()
   校验 -> defaults 合并 -> 记录参数来源

5. build_command_plan()
   把语义参数编译为锁定结构的 argv

6. ScopePolicy.decide()
   检查 target、active、requests/second、approval

7. CapsuleRunner.run()
   dry-run / fixture / live execution

8. RunResult + run.json
   保存 arguments、argument_sources、command 和证据

9. ObservationPacket
   只把受预算约束的结果交给模型
```

这个流程中，只有第 3 步具有模型不确定性。第 4 步之后都是确定性代码，因此大部分测试不需要真实 Agent。

## 4. `core/arguments.py`：参数协议的中心

### 4.1 `ResolvedArguments`

```python
@dataclass(frozen=True)
class ResolvedArguments:
    values: dict[str, Any]
    sources: dict[str, str]
```

`values` 是最终生效值；`sources` 解释每个值来自 Agent 还是 profile 默认值。它使用 `frozen=True`，避免解析完成后被其他层偷偷修改对象属性。

例子：Agent 只传 `depth=1`，profile 默认速率 10：

```json
{
  "values": {"depth": 1, "requests_per_second": 10},
  "sources": {"depth": "agent", "requests_per_second": "profile_default"}
}
```

审计来源很重要。看到最终 argv 中的 `10`，维护者能判断它是模型决定还是系统默认。

### 4.2 `resolve_arguments()`

函数输入：`Capsule`、profile 名、模型 provided dict。处理顺序不能随意颠倒：

1. 读取顶层 `input_schema.properties`。
2. 读取当前 profile 的 `defaults` 和 `allowed_arguments`。
3. 检查配置自身是否矛盾，例如 defaults 中出现没有 Schema 的字段。
4. 拒绝模型提供的未知字段。
5. 拒绝 Schema 存在但当前 profile 未开放的字段。
6. 检查 `x-agent-settable`。
7. `deepcopy(defaults)` 后用模型值覆盖。
8. 对最终值逐项执行 `validate_value()`。
9. 生成来源字典。

为什么使用 `deepcopy`：数组是可变对象。如果 Nuclei 默认 severity 列表被某次调用原地修改，后续调用可能继承污染后的默认值。深拷贝让每次解析拥有独立数据。

为什么越界时拒绝而不是 clamp：如果模型请求 100 RPS，Runtime 静默改为 5，Agent 可能误以为计划按 100 执行；审计也无法还原它的真实意图。明确错误更适合上层修复。

### 4.3 `validate_input_schema()`

这是 Capsule conformance 的入口，验证：

- Schema 顶层必须是 object。
- `additionalProperties` 必须是 false。
- property 必须是对象。
- 类型只能是 array/boolean/integer/number/string。
- 每个参数必须有 description。
- 每个 profile 默认值都能经过 `resolve_arguments()`。

项目没有声称实现完整 JSON Schema。只支持自己确实验证的子集，是比“接受任意 Schema 但运行时忽略关键约束”更可靠的选择。

### 4.4 `validate_value()`

这个函数递归验证单值。几个 Python 细节值得注意：

- `bool` 是 `int` 的子类，所以 integer 判断必须排除 `bool`。
- array 的每个 item 递归进入同一函数，Nuclei severity 因而能验证字符串枚举。
- `uniqueItems` 使用每项的 `repr()` 建立集合，足够覆盖当前标量数组，但不是完整 JSON 等价算法。
- 错误消息始终带参数名，便于 Agent 修复。

### 4.5 `template_value()`

Planner 最终需要字符串 argv：

- boolean 变成小写 `true/false`。
- array 变成逗号连接，例如 `['high', 'critical'] -> high,critical`。
- 数字和字符串用 `str()`。

它不做 shell escaping，因为每个结果仍是单独的 argv token，Runtime 不经过 shell。

## 5. `core/planner.py`：语义参数到 argv

新签名：

```python
build_command_plan(capsule, target, profile_name="safe", arguments=None)
```

Planner 先调用 `resolve_arguments()`，再建立模板变量：

```python
variables = {
    "target": target,
    "capsule_root": str(capsule.root),
    **validated_arguments,
}
```

它使用 `Template.substitute()`，不是旧版 `safe_substitute()`。区别是：

- strict substitute：缺变量立即报错。
- safe substitute：可能把 `$missing` 原样留在 argv。

对安全工具而言，带着残留模板启动程序是不确定行为，因此 strict 更合适。

`CommandPlan` 新增：

| 字段 | 用途 |
|---|---|
| `arguments` | 最终语义参数。 |
| `argument_sources` | 每项来源。 |
| `requests_per_second` | 给 Scope 使用的明确速率。 |
| `command` | 已编译 argv。 |

固定 flag 仍写在 YAML command 中。以 Katana 为例，Agent 可改变 depth 和速率，但不能移除 `-jsonl`、`-silent` 或把 `-fs fqdn` 改成跨域爬取。

## 6. `models.py`、`runner.py` 与 `scope.py`

### 6.1 RunResult 为什么重复保存 arguments 和 command

`arguments` 表达模型意图，`command` 表达工具实际收到的 argv。二者都保存才能完成审计：

```text
模型说了什么 -> Runtime 怎样解释 -> 外部程序实际执行什么
```

`RunResult.to_dict()` 将 arguments、argument_sources 一起写入 `run.json`。fixture 与 dry-run 也写，因此不用真实扫描就能检查参数计划。

### 6.2 Runner 的调用顺序

`CapsuleRunner.run()` 在创建子进程前：

1. `build_command_plan()`，参数错误在这里终止。
2. 加载 Scope。
3. 从 plan 取 `requests_per_second`。
4. `ScopePolicy.decide()` 比较 Scope 上限。
5. `decision.raise_if_denied()`。
6. 才进入 fixture、dry-run 或 live execution。

所以模型即使提交合法的 Capsule 参数 `requests_per_second=10`，Scope 最大值为 3 时仍会被策略层拒绝。Schema 是工具级绝对边界，Scope 是本次授权边界，两者不能合并成一个概念。

### 6.3 速率单位迁移

v0.1.1 的 Scope 字段叫 RPM，工具 flag 却按每秒解释。v0.1.2 使用 `max_requests_per_second`。

旧字段暂时按数学单位换算：

```text
max_requests_per_minute / 60 = max_requests_per_second
```

因此旧值 60 只允许 1 RPS。这个兼容入口计划在 v0.2 删除。

## 7. Registry：模型能看到什么

`capsule_to_public_dict()` 仍有三层：

| 层级 | 内容 |
|---|---|
| brief | ID、分类、stage、风险、摘要。 |
| usage | brief + best/avoid + input_schema + profile defaults/allowed arguments。 |
| full | 完整 YAML，包括 command 和 runtime。 |

模型通常不需要 full 就能生成参数。usage 已经回答“有哪些参数、类型、边界、默认值”。command 延迟到 full 才披露，减少上下文，也避免模型把工具 flag 当成应该直接返回的内容。

## 8. CLI、MCP 与 Recipe

### 8.1 CLI

CLI 使用 JSON 而不是自己拆 `key=value` 字符串：

```bash
sec-capsules plan katana \
  --target https://example.com \
  --arguments-json '{"depth":1,"requests_per_second":3}'
```

`parse_json_object()` 调用标准 `json.loads()`，并强制顶层为 object。Recipe 的 `parse_nested_json_object()` 进一步要求每个 step 的值也是 object。

JSON 适合复现 Agent 请求，也避免 `severity=high,critical` 到底是字符串还是数组的歧义。

### 8.2 MCP

MCP 仍只有七个元工具。`run_capsule()` 新增通用：

```python
arguments: dict[str, Any] | None
```

没有为 Katana/Nuclei 分别新增 MCP tool，因此工具数量增加时不会把所有扫描器定义同时塞进模型上下文。FastMCP 只负责协议适配，业务校验仍在 Core。

### 8.3 Recipe

Recipe 使用：

```json
{
  "arguments_by_step": {
    "crawl_endpoints": {"depth": 1},
    "baseline_scan": {"severity": ["critical"]}
  }
}
```

为什么按 step ID 而不是 Capsule ID：同一个 Recipe 未来可能两次调用同一 Capsule，但参数不同。`run_recipe()` 会在执行任何 step 前拒绝未知 step key，防止拼写错误被默默忽略。

Recipe 结果记录每步 arguments 和来源，但 recipe Observation 仍不复制这些大对象或 structured payload，继续保持上下文隔离。

## 9. 三个 Capsule 的开放参数

| Capsule/profile | Agent 可设置 | 固定不变量 |
|---|---|---|
| httpx safe | `requests_per_second` 1–20 | JSON、silent、单 target。 |
| katana safe | `depth` 1–5；RPS 1–10 | JSONL、silent、`fqdn` scope、单 target。 |
| nuclei safe | RPS 1–5；severity 只能是 medium/high/critical 的非空唯一数组 | JSONL、单 target、模板来源。 |
| nuclei local_lab | RPS 1–5 | 仓库内唯一 Juice Shop 模板路径。severity 不开放。 |

`local_lab` 能说明 Schema 和 profile 是两级门槛：severity 在 Nuclei Schema 中，但该 profile 没有放进 allowed list，所以 Agent 不能借 local_lab 改成通用扫描。

## 10. 为什么模块不需要完整 Agent 也能测试

一次插件调用的边界是结构化输入和结构化输出。测试只需扮演调用者：

```text
测试提供 InvocationRequest
  -> Core 生成/拒绝 CommandPlan
  -> fixture 或本地靶场产生输出
  -> Core 返回 RunResult/Observation
```

Agent 的记忆、计划循环、反思和 UI 都不属于这个契约。为测试它们而引入 LangChain/CrewAI 等框架，反而会让失败来源变得模糊。

## 11. 测试分层

### 11.1 PR CI：确定性测试

当前 41 项 unittest 覆盖：

| 文件 | 主要契约 |
|---|---|
| `test_arguments.py` | 默认值、Agent 覆盖、类型、范围、枚举、重复数组、profile 隔离。 |
| `test_capsule_conformance.py` | 每张 Capsule/每个 profile 的 Schema 与默认计划。 |
| `test_runner.py` | 参数进入 Scope、RunResult 和 run 文件。 |
| `test_cli.py` | JSON 对象边界。 |
| `test_mcp_server.py` | MCP arguments 穿透和 execution gate。 |
| `test_recipe.py` | step 参数和未知 step 拒绝。 |
| `test_evals.py` | 参考候选、错误候选、上下文指标和基准报告。 |
| 其余旧测试 | executor、artifact、parser、Observation、Scope。 |

这些测试不调用模型、不访问公网，适合作为每次 PR 的硬门槛。

### 11.2 手动本地 E2E

`scripts/e2e-local.sh` 继续在仅绑定 127.0.0.1 的 Juice Shop 上运行真实 httpx、katana、nuclei。它验证外部二进制兼容性，不评价模型聪明程度。

### 11.3 Agent Eval

真实模型可能随版本、温度和服务状态变化，所以放在手动/nightly 层。外部调用者把场景和 `get_capsule` 输出交给任意模型，再把模型返回保存为 candidate JSON；Harness 负责统一评分。

## 12. `evals/harness.py` 逐函数说明

### 12.1 `load_data()`

按扩展名读取 JSON 或 YAML，强制顶层 object。场景适合 YAML（人写），模型候选适合 JSON（机器生成）。

### 12.2 `grade_candidate()`

评分项：

- capsule 是否在 available list。
- capsule/profile 是否符合 expected。
- arguments 是否为对象。
- 能否经过真实 Planner。
- 是否满足 equals/minimum/maximum/one_of/contains_all 等场景约束。
- forbidden 参数是否缺席。
- candidate 是否只提供任务需要的参数，作为最小权限检查。

错误模型输出不会启动工具。Planner 异常被保存到 `errors`，结果标记失败。约束比较也捕获类型不匹配，避免评分器自身崩溃。

这里的 expected 是测试隐藏答案，不交给模型。

### 12.3 `measure_disclosure()`

对照两种上下文：

```text
raw_full：一次给出所有候选 Capsule 的完整 YAML
progressive：所有 brief + 选中 Capsule 的 usage
```

使用 `characters / 4` 粗略估 token。它只适合比较同一仓库版本的趋势，不是 OpenAI/Anthropic 等供应商的精确计费器。

当前两个参考场景在本机得到：

| 场景 | full 估算 | progressive 估算 | 比例 |
|---|---:|---:|---:|
| Katana shallow | 1409 | 512 | 36.34% |
| Nuclei critical | 1409 | 613 | 43.51% |

这些数字随工具卡内容变化，应由 CI/Eval 重新生成，不应手工当作永久承诺。

### 12.4 `benchmark_planner()`

重复调用真实 Planner，报告 mean、p95、min、max。当前开发机 1000 次 Katana 样本约为平均 0.028 ms、p95 0.038 ms。

CI 只检查函数能产生报告，不限制固定毫秒值。共享 runner 的负载会波动，硬阈值容易制造假失败；发布评估应看多次趋势。

## 13. 怎样接入一个真实模型而不污染 Core

模型适配器只需要完成：

```text
scenario.objective + Capsule brief/usage
  -> provider API
  -> 保存 InvocationRequest JSON
  -> grade_candidate()
```

它可以放在独立脚本或后续 `evals/adapters/`，API key 由环境提供。Core 不 import 模型 SDK，也不认识 prompt、temperature 或 provider。

记录真实 Eval 时至少保存：模型标识、prompt 版本、调用时间、输入/输出 token、候选 JSON、评分结果、修复次数。不要只保存“通过率”，否则无法解释回归来源。

## 14. 新增一个参数的实战步骤

以给 Katana 新增合法的 timeout 参数为例（这里只说明流程，不代表当前已经开放该工具 flag）：

1. 查官方工具文档，确认参数语义和单位。
2. 在 `input_schema.properties` 添加类型、description、绝对范围。
3. 在需要开放的 profile 增加 defaults 与 allowed argument。
4. 在 command 中增加固定 flag 和 `$variable`。
5. 在 `test_arguments.py` 写默认、合法覆盖、边界和错误类型测试。
6. 在 conformance test 下编译所有 profile。
7. 用 `sec-capsules plan` 人工核对 argv。
8. 若影响 Scope/审批，增加策略测试。
9. 若影响真实工具兼容，更新本地 E2E。
10. 更新 Capsule Spec 和 usage 示例。

不要从“先让模型传 `extra_args`，以后再规范”开始。那会绕过本版本建立的整个知识和策略边界。

## 15. 当前限制与 v0.2 接口

- Schema validator 是明确的子集，还没有 `$ref`、`oneOf`、条件 Schema。
- 没有凭据引用类型，不能安全承载认证扫描参数。
- Eval Harness 目前是单轮候选评分，没有 provider adapter 和多轮 repair 统计。
- context token 是粗估，不是供应商 tokenizer。
- Planner 基准只测本模块开销，不测外部扫描器速度。
- Scope 仍不是网络代理，不能拦截工具内部所有派生请求。
- v0.1.1 通用 redaction、审批签名等限制仍然存在。

v0.2 新增 Nmap/FFUF 时，首先复用这套参数协议，而不是为每个工具新增专属 MCP tool。Nmap 将验证非 JSONL 服务参数，FFUF 将验证高噪声参数和最小权限；它们也会扩充 Eval 场景，证明当前抽象具有通用性。

## 16. 最后记住的设计句子

```text
Profile 是安全预设，不是固定死命令。
Agent 选择语义参数，不拼 argv。
Schema 定义工具绝对边界，Scope 定义本次授权边界。
Eval Harness 测试一次插件契约，不冒充完整 Agent。
```

理解这四句话，就掌握了 v0.1.2 最重要的代码思想。
