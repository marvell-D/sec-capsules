# v0.2.0a2 Agent 基准开发者手册

这份手册解释 v0.2.0a2 为什么新增 FFUF，Agent A/B 基准怎样运行，每个主要文件和函数负责什么，以及应该怎样解读实验结果。它的目标是让维护者能够自己修改代码，而不是只会执行一条命令。

## 1. 版本要证明什么

v0.1.2 已经证明 Agent 可以提交受 JSON Schema 约束的语义参数，v0.2.0a1 用 Nmap 证明 Capsule 不只能表示 HTTP endpoint。但这两版还不能回答一个核心问题：

> 在多轮 Agent 任务中，Capsule 相对“模型自己写 argv，然后看全部 stdout”究竟带来了什么？

v0.2.0a2 用两个切片回答：

1. FFUF 产生高基数 JSONL，检验去重、计数、证据引用和上下文预算。
2. Reference Agent 保持任务、模型、轮数和评分不变，只替换 Raw/Capsule Adapter。

这个版本仍然不是自主渗透框架。`benchmarks/` 位于项目顶层，不在 `src/` 内，因此不会被 setuptools 打入 `sec-capsules` wheel。它是 Runtime 的测试消费者。

## 3. FFUF Capsule

### 3.1 参数为什么只有两个

`capsule.yml` 只允许 Agent 提交：

- `match_status: list[int]`：1 到 10 个唯一 HTTP 状态码，每个值在 100 到 599 之间。
- `requests_per_second: int`：1 到 50，同时声明速率单位为 `requests_per_second`。

字典路径、线程数 5、请求超时 5 秒、总运行上限 60 秒、JSONL 输出和静默模式都是 profile 固定命令片段。这样设计的原因是：模型应该做“在什么强度下保留哪些结果”的业务决策，不应该拥有任意字典路径、recursion 或额外 flag 注入能力。

`safe` profile 是 active 行为，action 为 `content_discovery`，且始终 `requires_approval: true`。Scope 还会再比较请求速率。两层约束的职责不同：Capsule 定义工具允许的上限，Scope 定义这次授权允许的上限。

### 3.2 parser 的数据结构

`parse(raw_text, run_id, artifact_name)` 逐行读取 FFUF JSONL，最终返回 Core 要求的五个集合，加一个可选诊断对象：

```python
{
    "assets": [],
    "services": [],
    "endpoints": [...],
    "findings": [],
    "evidence": [...],
    "parse_diagnostics": {
        "input_records": 18,
        "parsed_records": 16,
        "invalid_records": 2,
        "duplicate_records": 1,
        "emitted_records": 15,
        "partial": True,
    },
}
```

主要函数：

- `parse()`：主流程，负责逐行 JSON 解码、验证 URL、去重、生成 endpoint/evidence 和诊断计数。
- `_optional_int()`：只接受真正的 `int`，不把布尔值或字符串暗中转换为数字。parser 对工具版本漂移采用保守策略。
- `_copy_optional()`：只在值存在时写入可选字段，避免制造大量 `null`。

去重键为 `(url, method, status, content_length)`。只用 URL 会丢掉同一路径不同方法或响应的差异；将 words/lines 也加入又会对微小页面波动过于敏感。当前键是一个明确可测的折中。

endpoint 可选字段包括 `status_code`、`content_length`、`content_type`、`words`、`lines`、`redirect_location` 和 `fuzz_input`。每个 endpoint 都持有一个指向原始 JSONL 行的 `artifact://...#L<n>` 证据引用。

## 4. ObservationPacket 的可见性统计

`build_observation_packet()` 新增 `result_counts`：

```json
{
  "endpoints": {"total": 30, "retained": 12, "omitted": 18}
}
```

`total` 来自 parser 输出，`retained` 是最终真正进入 Observation 的数量，`omitted = total - retained`。这避免 Agent 把“看到 12 条”误解为“工具只找到 12 条”。

相关私有函数：

- `_refresh_result_counts()`：每次因 token budget 弹出一个对象后，重新计算 retained/omitted。
- `_estimated_with_budget()`：在真正加入 `budget` 字段前就把该字段的大小算进预算，避免“剪枝时没超标，加上预算说明后又超标”。

剪枝顺序是 endpoint、service、asset、finding 的尾部元素，然后是空集合键、parser diagnostics、next action 和 execution 摘要。在 120-token 这种极小预算下，会保留结果总数、raw output 隐藏标记和脱敏状态，先省略可推导的 drill-down 标志。

## 5. Reference Agent

### 5.1 为什么 Agent 必须很薄

`run_reference_agent()` 不做领域规划，不内置漏洞知识，不使用外部搜索。它只做四件事：

1. 将场景的公开部分和 Adapter 使用说明交给模型。
2. 接收模型的 JSON tool/final 动作。
3. 把 tool 动作转给 Adapter，再把 Adapter 输出放回对话。
4. 记录 trace 和 provider usage。

模型协议只有两种 JSON：

```json
{"type":"tool","tool":"run_capsule","input":{}}
```

```json
{"type":"final","answer":{"assets":[],"services":[],"endpoints":[],"findings":[],"evidence_refs":[]}}
```

如果模型返回其他形状，trace 终止为 `invalid_model_action`；如果用完轮数仍未 final，终止为 `turn_limit`。这些状态不会被报告层伪装成成功。

### 5.2 每轮记录什么

Agent 在调用 provider 前序列化当前 messages，记录：

- `context_bytes`：该轮输入上下文的 UTF-8 字节数。
- `messages_sha256`：该轮 messages 的摘要，用于检查对比输入是否意外变化。
- `response`：模型 JSON 动作。
- `usage`：provider 原始 token usage。
- `latency_ms` 和 provider trace id。

`_sum_usage()` 兼容 `prompt_tokens/completion_tokens` 和 `input_tokens/output_tokens` 命名，并将 cache hit token 单独汇总。真实 token 对比必须用这些 provider 字段；字符数除以 4 只能作为 Runtime 的预算近似，不能用于宣称模型 token 节省。

## 6. 两个 Adapter

### 6.1 RawToolAdapter

Raw 组模拟常见的薄 Agent 工具集成：模型生成 argv，模型看到全部 stdout/stderr。但基准不会为了“普通”而放弃最基本的实验安全：

- argv 必须是字符串数组，通过 `subprocess` 执行，不经 shell。
- 只允许 Nmap/FFUF 的明确 flag 子集。
- Nmap 只能使用场景 host，端口数、范围、速率、超时和 XML stdout 都受检查。
- FFUF 必须使用打包字典和场景 URL，线程/超时/总时间固定。
- 两个工具都经过同一 Scope/approval 授权。

这些防护不是 Capsule 优势的一部分，而是基准自身的安全边界。Raw 组仍然没有语义参数 Schema、渐进工具卡、统一 parser 和 Observation 压缩。

`_parse_options()` 是一个小型确定性 argv parser：它区分无值 flag、带值 flag 和位置参数，拒绝未知或重复 flag。`_bounded_int()` 处理速率上限。它们不是通用 shell parser，只是这个受控基准的 allowlist。

### 6.2 CapsuleToolAdapter

Capsule 组暴露三个元工具：

- `search_capsules`：搜索 brief 卡，不传输参数 Schema。
- `get_capsule`：只取某一 Capsule 的 usage 卡。
- `run_capsule`：提交 capsule id、target、profile 和语义 arguments。

Adapter 内部调用真正的 `CapsuleRunner`。Replay 时传入 fixture，Live 时传入 `execute=True`。它只把 `result.observation` 返回模型，但 trace 仍会记录 artifact path、raw bytes、run id 和 Scope decision。因此“模型没看到 raw”不等于“证据被丢弃”。

`AdapterResult` 是两组共用的交换结构：

```python
AdapterResult(
    status="replayed",
    model_output={...},
    raw_output_bytes=123456,
    model_visible_bytes=2600,
    duration_ms=12.3,
    run_id="run_xxx",
    artifact_refs=[...],
    policy={...},
)
```

这个结构将“工具真正产生多少”和“模型实际看到多少”分开。

## 7. Replay 与 Live

### 7.1 Replay

Replay 不运行 Nmap/FFUF。两组读取同一 fixture：

- Raw 组将 fixture 原文作为 stdout 交给模型。
- Capsule 组让 Runner 解析 fixture，只交 ObservationPacket。

这层去掉了网络、工具版本、目标时序和系统负载的波动，适合比较上下文负担和模型整理结果的能力。

### 7.2 Live

Live 调用 `scripts/benchmark-agent.sh live`，脚本会：

1. 检查 Docker、Nmap 和 FFUF。
2. clone/fetch 锁定到固定 commit 的 OWASP crAPI。
3. 将服务绑定到 `127.0.0.1`并等待 health endpoint。
4. 运行成对的 raw/capsule 实验。
5. 无论成功失败都删除该 compose project 的容器和 volume。

Live 能测量真实运行时间、工具失败、parser 版本适配和结果波动。它不如 Replay 可重复，因此两层都需要，不应该互相取代。

## 8. 场景与隐藏评分

`crapi-recon-replay.yml` 含有：

- `task` 和 `targets`：会交给 Agent。
- `available_capsules`：Capsule Adapter 的可用集合。
- `scope_file` / `approval_file`：运行时授权。
- `fixtures`：Replay 数据。
- `limits`：轮数、model output、Observation、tool timeout 和 raw output 上限。
- `milestones`：只交给评分器，不交给 Agent。

`score_trace()` 支持的第一批 milestone：

- `asset_value_contains`：最终 asset 是否包含期望 host。
- `service_ports_contains`：服务端口是否覆盖期望集合。
- `endpoint_urls_contains`：endpoint URL 是否覆盖期望集合。
- `evidence_refs_min`：是否保留足够证据引用。

评分不请另一个 LLM 做 judge，因为那会引入额外的模型偏差、token 成本和不稳定性。

## 9. agent_trace.v1 与报告

trace 顶层包含：

- 身份：`trace_id`、`scenario_id`、`variant`、`provider`、`model`。
- 时间与状态：`started_at`、`finished_at`、`status`。
- 详细过程：`model_calls`、`tool_calls`、`final_answer`。
- 客观结果：`score`。
- 汇总指标：`totals`。

`totals` 中最重要的指标：

- `input_tokens/output_tokens/total_tokens/cached_tokens`：provider 报告的真实用量。
- `raw_tool_output_bytes`：工具 stdout/stderr 实际字节数。
- `model_visible_tool_bytes`：Adapter 实际返回对话的 JSON 字节数。
- `peak_context_bytes`：最大单轮上下文字节数。
- `tool_failures`：工具失败或拒绝总数。
- `policy_denials`：其中由安全策略明确拒绝的数量。
- `wall_time_ms`：整条 Agent trace 耗时。

`build_report()` 按 variant 分组，对 token、bytes、上下文和耗时取中位数，对 milestone score 取平均，对失败/拒绝求和。中位数比平均值更不容易被某一次 API 拖延或工具异常拉偏。

不要只看 token 一个数。正确解读应同时考虑：

1. milestone score 是否下降。
2. evidence coverage 是否下降。
3. policy denial 是模型越权，还是场景授权配置错误。
4. 小输出时 Capsule 的工具卡和结构化字段可能比 raw 更大。
5. 输出增长后，Capsule 可见上下文是否保持有界。

“找到临界点”比“所有场景都宣称节省”更有工程价值。

## 10. 测试策略

`tests/test_agent_benchmark.py` 的四个测试分别覆盖：

1. **同一 milestone**：ScriptedProvider 驱动 raw/capsule 两组，两组都必须通过四个隐藏 milestone。
2. **高基数压力**：将 FFUF fixture 重复 100 次，产生 1800 条 raw record。由于内容重复，parser 仍只输出 15 个 endpoint；断言 Capsule 可见字节小于 Raw 的 10%。
3. **安全拒绝**：Raw Agent 尝试 `nmap --script`，Adapter 必须在执行前拒绝，并产生 `policy.allowed=false`。
4. **报告口径**：向 ScriptedProvider 注入确定 token usage，验证 trace/report 使用 provider 数据，同时检查 trace 必要字段。

`ScriptedProvider` 不是一个假 Agent。它的用途是把模型随机性从 CI 中移除，让我们能稳定检查 Adapter、trace、scoring 和 reporting 的代码。真实模型实验是另一层，不应该取代确定性 CI。

## 11. 怎样运行

本地全部确定性测试：

```bash
python -m pip install -e .
scripts/ci.sh
```

真实模型 Replay：

```bash
export SILICONFLOW_API_KEY='a-new-rotated-secret'
scripts/benchmark-agent.sh replay 3
```

指定精确模型：

```bash
scripts/benchmark-agent.sh replay 5 'exact/model-id-from-account-list'
```

自有 runner 上的真实工具实验：

```bash
scripts/benchmark-agent.sh live 3 'exact/model-id-from-account-list'
```

也可以在 GitHub Actions 手动触发 `Agent Benchmark`，选择 replay/live、repeats 和 model。密钥必须位于 repository secret `SILICONFLOW_API_KEY`，workflow 不接受密钥 input。

产物位于：

```text
benchmark-reports/<mode>/
├── trace_xxx-raw-r1.json

```

## 12. CI/CD 到底做了什么

CI 是 Continuous Integration，即每次 push/PR 都在干净环境中重新验证代码能否编译、所有契约是否通过。本项目的 `scripts/ci.sh` 做两件事：

1. `compileall src benchmarks tests`：在测试前发现语法错误和无法导入的模块。
2. `unittest discover`：运行 Core、Capsule、parser、Scope、MCP、Eval 和 Agent benchmark 的确定性测试。

CD 在这个 pre-release 项目中不是“每次 push 都自动扫描”。安全工具和真实模型都有授权、成本和环境状态，因此它们是 `workflow_dispatch` 手动 workflow。workflow 会上传 trace/report artifact，而不是自动将扫描结果发布到公开网络。

## 13. 当前局限和下一步

- 当前 milestone 只覆盖 recon-to-handoff，不测试漏洞利用。
- FFUF 字典故意很小，Live 实验的目标是工具调用质量，不是最大化覆盖率。
- Raw Adapter 是一个受控 baseline，不代表所有现有 Agent 框架。后续应增加可插拔的外部 Agent adapter，而不把它们写入 Core。
- 当前报告是按 variant 聚合，还可增加按 repeat index 的严格成对差值和置信区间。
- 真实 token 结论至少应运行 5 组 Replay 和 3 组 Live，报告模型精确 ID、日期、场景 commit 和方差。
- 应继续测量 break-even point：从小输出逐步放大 artifact，找到 Capsule 总上下文开始小于 Raw 的规模，而不只保留 100 倍极端点。

下一版最值得的优化不是再加一个 Agent 功能，而是让实验更严谨：严格 paired report、多个输出规模点、真实模型方差和一个外部公开 benchmark 子集。
